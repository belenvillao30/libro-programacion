3.1

Los Tres Tipos de Error en Programación

Todo programador comete errores. La diferencia entre un principiante y un programador competente no es que el segundo no los cometa — es que los identifica y corrige con método. El primer paso es saber con qué tipo de error se está tratando.

🔴
Error de Sintaxis
El código viola las reglas gramaticales del lenguaje. El compilador no puede traducirlo.
Detectado por: compilador
🟡
Error Lógico
El código compila y ejecuta, pero produce resultados incorrectos. El algoritmo tiene un fallo.
Detectado por: el programador
🔵
Error de Ejecución
El programa compila pero falla en tiempo de ejecución: división por cero, acceso a memoria inválida.
Detectado por: el sistema
🤖 IA y los errores lógicos

El compilador detecta errores de sintaxis automáticamente. La IA también puede sugerirte correcciones de sintaxis. Pero los errores lógicos son invisibles para ambos — un programa que calcula mal el IVA porque usó la fórmula equivocada compila perfectamente. Solo quien entiende el problema puede detectarlo.

3.2

Errores de Sintaxis: Leer al Compilador

El compilador es tu aliado, no tu enemigo. Sus mensajes de error contienen información precisa: archivo, número de línea, descripción del problema. Aprender a leerlos es una habilidad fundamental.

Errores de sintaxis más frecuentes en C++

ErrorCausaMensaje típico del compilador

{ }Llave de cierre faltante o extraexpected '}' at end of input

Cómo leer un mensaje de error real

C++ · Con error de sintaxis
con_error.cpp 
#include <iostream>
using namespace std;

int main() {
    double precio = 25.50
    double iva = precio * 0.12;
    cout << "IVA: " << iva << endl
    return 0;
}
con_error.cpp:5:28: error: expected ';' before 'double'
5 | double precio = 25.50
| ^
| ;
con_error.cpp:8:35: error: expected ';' before 'return'
💡 El compilador señala la LÍNEA y el CARÁCTER exacto. Siempre empieza por el primer error — corregirlo puede eliminar los siguientes.
📌 Regla del primer error

Un solo error de sintaxis puede generar una cascada de 5 o 10 mensajes. Siempre corrige el primer error de la lista y vuelve a compilar. Los errores posteriores suelen desaparecer.

El mismo código corregido

🚫 Con errores
double precio = 25.50   // falta ;
cout << "IVA: " << iva << endl  // falta ;
✅ Corregido
double precio = 25.50;  // ; al final
cout << "IVA: " << iva << endl;  // ; al final
3.3

Errores Lógicos: Los Más Peligrosos

Un error lógico es aquel que el compilador no puede ver. El programa compila, ejecuta, muestra resultados — pero esos resultados son incorrectos. Solo el programador que comprende el problema puede detectarlos.

Errores lógicos frecuentes en esta etapa

🚫 Error lógico — División entera
int a = 7, b = 2;
double resultado = a / b;
// resultado = 3.0 (no 3.5)
// El compilador no dice nada
✅ Correcto
int a = 7, b = 2;
double resultado = (double)a / b;
// resultado = 3.5 ✓
🚫 Error lógico — Fórmula equivocada
// Promedio ponderado incorrecto
double prom = nota1 + nota2 * 0.5;
// Se aplica 50% solo a nota2
// por la precedencia de operadores
✅ Correcto
// Paréntesis garantizan el orden
double prom = (nota1 + nota2) * 0.5;
// Ambas notas tienen igual peso
🚫 Error lógico — = en lugar de ==
int x = 5;
if (x = 10) {   // asigna 10 a x
    cout << "es diez";
}
// Siempre verdadero: x ahora es 10
✅ Correcto
int x = 5;
if (x == 10) {  // compara x con 10
    cout << "es diez";
}
// Solo entra si x vale 10
⚠️ La trampa de la IA con errores lógicos

Si le muestras a la IA un código con error lógico y le preguntas "¿funciona bien este código?", a menudo dirá que sí — porque compila. La IA no conoce tus datos de prueba ni el resultado esperado. La verificación lógica es siempre responsabilidad del programador.

3.4

Proceso Sistemático de Depuración

Depurar sin método lleva a frustraciones innecesarias. Un proceso sistemático convierte la depuración en una habilidad transferible. El mismo proceso se aplica en todos los lenguajes y a lo largo de toda la carrera.

1
Reproducir el error
Identifica con qué datos de entrada ocurre el problema. Un error que no se puede reproducir no se puede corregir. Documenta los valores exactos que lo provocan.
2
Leer el mensaje de error completo
Si es un error de compilación, lee el mensaje entero antes de cambiar el código. Identifica: archivo, línea, tipo de error. No adivines — el compilador ya te dice dónde está.
3
Aislar con tabla de simulación
Para errores lógicos: traza el algoritmo paso a paso en papel con los valores que fallan. Anota el valor de cada variable en cada paso. El error aparece donde el valor esperado y el real divergen.
4
Insertar cout de diagnóstico
Agrega instrucciones cout temporales que muestren el valor de las variables en puntos clave. Esto confirma dónde se rompe la lógica sin necesidad de un debugger externo.
5
Corregir y verificar
Aplica un cambio a la vez. Verifica que el error desapareció con los mismos datos de entrada. Luego prueba con datos extremos: cero, negativos, valores muy grandes.
6
Eliminar los cout de diagnóstico
Antes de entregar, elimina todos los cout de depuración que agregaste. El código final solo debe mostrar la salida requerida por el problema.
C++ · Técnica de cout diagnóstico
depuracion_ejemplo.cpp 
double precio = 50.0;
int    cantidad = 3;
double total;

// DEPURACIÓN: verificar valores antes del cálculo
cout << "[DEBUG] precio=" << precio << " cantidad=" << cantidad << endl;

total = precio * cantidad;

// DEPURACIÓN: verificar resultado
cout << "[DEBUG] total=" << total << endl;

// Una vez verificado, eliminar los cout de DEBUG antes de entregar
3.5

Buenas Prácticas de Programación

El código se escribe una vez, pero se lee muchas. Un programa que solo tú entiendes hoy, en dos semanas será incomprensible incluso para ti. Las buenas prácticas no son estética — son responsabilidad profesional.

1. Comentarios que explican el por qué, no el qué

🚫 Comentario inútil
// Multiplicar precio por cantidad
total = precio * cantidad;

// Dividir entre 100
descuento = pct / 100;
✅ Comentario útil
// Calcular subtotal antes de impuestos
total = precio * cantidad;

// Convertir porcentaje a fracción decimal
descuento = pct / 100.0;

2. Indentación y formato consistentes

🚫 Sin indentación
int main(){
double x=5;
if(x>0){
cout<<"positivo";
}
return 0;}
✅ Indentación correcta
int main() {
    double x = 5;
    if (x > 0) {
        cout << "positivo";
    }
    return 0;
}

3. Nombres significativos

🚫 Nombres crípticos
double a, b, c, d;
a = 1250.0;
b = a * 0.12;
c = a * 0.095;
d = a - b - c;
✅ Nombres descriptivos
double sueldoBruto    = 1250.0;
double impuestoRenta  = sueldoBruto * 0.12;
double aporteIESS     = sueldoBruto * 0.095;
double sueldoNeto     = sueldoBruto - impuestoRenta - aporteIESS;

4. Una instrucción por línea y espacios legibles

🚫 Todo comprimido
int a=5,b=10,c=15;double p=(a+b+c)/3.0;cout<<p;
✅ Claro y espaciado
int nota1 = 5, nota2 = 10, nota3 = 15;
double promedio = (nota1 + nota2 + nota3) / 3.0;
cout << "Promedio: " << promedio << endl;

5. Estructura de un programa completo bien comentado

C++ · Ejemplo de buenas prácticas
buenas_practicas.cpp 
/*
 * Programa: Calculadora de nota final
 * Descripción: Calcula la nota final ponderada de un estudiante
 * Autor: [Nombre del estudiante]
 * Fecha: [Fecha]
 */
#include <iostream>
#include <iomanip>
using namespace std;

int main() {
    // --- Pesos por componente (deben sumar 1.0) ---
    const double PESO_EXAMENES   = 0.40;
    const double PESO_PRACTICAS  = 0.35;
    const double PESO_TAREAS     = 0.25;

    // --- Variables de entrada ---
    double notaExamenes, notaPracticas, notaTareas;

    // --- Solicitar calificaciones ---
    cout << "Nota de exámenes (40%):  ";
    cin  >> notaExamenes;
    cout << "Nota de prácticas (35%): ";
    cin  >> notaPracticas;
    cout << "Nota de tareas (25%):    ";
    cin  >> notaTareas;

    // --- Calcular nota ponderada ---
    double notaFinal = notaExamenes  * PESO_EXAMENES
                     + notaPracticas * PESO_PRACTICAS
                     + notaTareas    * PESO_TAREAS;

    // --- Mostrar resultado formateado ---
    cout << fixed << setprecision(2);
    cout << "\nNota final: " << notaFinal << "/10" << endl;

    return 0;
}

Checklist de buenas prácticas — usar antes de cada entrega

✅ Checklist de entrega — Cap. 3
Encabezado de programa Incluye: nombre del programa, descripción breve, autor, fecha.
Variables declaradas con nombre significativo No hay variables de un solo carácter (excepto contadores i, j).
Constantes declaradas con const Ningún valor literal "mágico" sin nombre en el código.
Indentación consistente 4 espacios por nivel. Todas las llaves alineadas.
Comentarios en secciones Al menos: entrada, proceso y salida comentadas.
Sin cout de depuración Eliminar todos los [DEBUG] antes de entregar.
Salida formateada Números con fixed + setprecision donde corresponde.
Probado con casos extremos Al menos: valores normales, cero, y un valor negativo si aplica.
3.6

Ejercicios de Aplicación

✏️ Actividades de Aplicación
1
Comprensión
Clasificar errores. Indica si cada error es de sintaxis, lógico o de ejecución, y justifica: (a) cout << "Hola" sin punto y coma. (b) Calcular promedio como suma / 0. (c) Usar = en lugar de == en una condición. (d) Olvidar #include <iostream>. (e) Dividir dos enteros esperando un decimal.
2
Comprensión
Leer mensajes del compilador. Compila el siguiente código y lee el mensaje de error completo: int main() { double x = 5.0 cout << x; return 0; }. Identifica: línea del error, carácter exacto, mensaje. Corrígelo y vuelve a compilar.
3
Aplicación
Depurar con tabla de simulación. El siguiente programa calcula el área de un trapecio pero da un resultado incorrecto. Traza la ejecución paso a paso, identifica el error y corrígelo: double b1=8, b2=5, h=4; double area = b1 + b2 * h / 2; El resultado esperado es 26.
4
Aplicación
Refactorizar con buenas prácticas. El siguiente código funciona pero viola todas las buenas prácticas. Reescríbelo respetando el checklist del §3.5: int main(){double a=1500,b=a*0.12,c=a*0.095,d=a-b-c;cout<<d;}
5
Integrador + IA
Auditoría de calidad. Toma tu solución del ejercicio evaluativo del Cap. 2 y evalúala con el checklist del §3.5. Luego pide a la IA que evalúe el mismo código con el criterio: "¿Este código C++ sigue buenas prácticas de programación? Lista los problemas que encuentras." Compara: ¿detectó los mismos problemas que tú? ¿Detectó algo que tú no viste? ¿Pasó algo por alto?
Actividad Presencial Evaluativa — Capítulo 3 · Cierre de Unidad I
Sesión 3 · Tiempo: 50 min · Sin acceso a IA
Se entrega un programa C++ con 3 errores de sintaxis y 2 errores lógicos (el docente lo imprime). El estudiante debe: (1) identificar y corregir todos los errores en papel, (2) justificar por escrito cada corrección, (3) implementar la versión corregida aplicando todas las buenas prácticas del §3.5, incluyendo el encabezado de programa y comentarios de sección.
  • Los 3 errores de sintaxis identificados correctamente con su línea.
  • Los 2 errores lógicos identificados con explicación del comportamiento incorrecto.
  • Código final cumple todas las condiciones del checklist de entrega.
  • El estudiante puede explicar cada corrección al ser consultado oralmente.
  • Prueba con valores de verificación proporcionados por el docente.
