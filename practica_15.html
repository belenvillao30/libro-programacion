🏆 Unidad IV · Sesión Práctica 15 — Proyecto Final

Sistema de Gestión Académica ESPAM MFL

Proyecto integrador · todos los conceptos del curso · 3 sesiones · entrega con defensa oral

Duración total3 × 180 min
ModalidadIndividual + defensa oral
Entregable.cpp + documentación + demo
F1 · Diseño F2 · Módulo Datos F3 · Módulo Notas F4 · Reportes F5 · Integración F6 · Defensa
Objetivo del Proyecto Final
El estudiante diseña, implementa, prueba y defiende oralmente un Sistema de Gestión Académica en C++ que integra: arreglos paralelos, matrices 2D, funciones con paso por valor/referencia/const&, validación de entradas, algoritmos de búsqueda y ordenamiento, procesamiento de cadenas y generación de reportes estadísticos con visualización ASCII. El sistema debe compilar sin errores y el estudiante debe poder explicar cualquier línea.
🎓 Contexto y Alcance del Proyecto
Desarrollar un sistema de consola completo para la carrera de Computación de ESPAM MFL con capacidad para 30 estudiantes, 4 notas por estudiante, 5 módulos funcionales (Estudiantes, Notas, Ordenamiento, Reportes, Menú), y entrega en tres sesiones presenciales. La sesión de defensa oral evalúa que el código sea original y comprendido.
F1
Sesión 1 — Diseño de Arquitectura en Papel
Sesión 1 · 0 – 60 min · SIN computadora

El sistema se diseña antes de escribir código. Los proyectos que empiezan en el IDE sin diseño terminan reescribiéndose. Dedica la primera hora completamente al diseño en papel.

1
Diagrama de módulos y dependencias
Dibuja los 5 módulos (Menú, Estudiantes, Notas, Ordenamiento, Reportes) como cajas. Traza flechas indicando cuál módulo llama a funciones de otro. Identifica qué módulos son "de servicio" (utilitarios que todos usan) y cuáles son de nivel superior. Escribe al menos 3 funciones dentro de cada caja con su firma aproximada.
2
Diseño de arreglos paralelos
Dibuja los 6 arreglos globales: cedulas[30], nombres[30], notas[30][4], promedios[30], activo[30], totalEst. Para cada uno especifica: tipo de dato, capacidad y propósito. Dibuja un ejemplo con 3 estudiantes reales mostrando cómo el índice i conecta todos sus datos.
3
Tabla de prototipos y decisiones de paso
Para cada función del catálogo del Cap. 15, completa una tabla con: nombre, parámetros con tipo de paso (valor/referencia/const&), tipo de retorno y justificación de cada decisión. Mínimo 12 funciones. Esta tabla es el contrato del sistema y parte de la entrega evaluada.
4
Casos de prueba del sistema completo
Define 5 estudiantes con nombres y cédulas reales (ficticias pero válidas, 10 dígitos). Calcula a mano sus promedios y clasificación. Estos son los datos con los que demostrarás el sistema en la defensa. Anota también los casos límite: ¿qué pasa si el sistema está lleno? ¿si buscas una cédula inexistente? ¿si todos repueban?
⚠️ Regla de oro: el diseño en papel se entrega junto al código

El diagrama de módulos, la tabla de prototipos y los casos de prueba son parte obligatoria de la entrega. Un proyecto sin documentación de diseño, aunque compile perfectamente, no refleja el proceso de ingeniería que se evalúa. Fotografía o escanea tu trabajo en papel antes de la siguiente sesión.

F2
Sesión 1 — Módulo de Datos y Estudiantes (60–180 min)
Sesión 1 · 60 – 180 min

Implementa primero la base: las estructuras de datos y el módulo de estudiantes. Verifica cada función antes de continuar.

C++ · Cabecera del archivo — variables globales y prototipos primeros
/* ═══════════════════════════════════════════════════════ Sistema de Gestión Académica — ESPAM MFL Autor : [tu nombre completo] NRC : [tu sección] Fecha : [fecha de entrega] Descripción : Sistema de consola para gestión de estudiantes y calificaciones de la carrera de Computación. Compilar con: g++ -Wall -Wextra -o sistema gestion.cpp ═══════════════════════════════════════════════════════ */ #include <iostream> #include <iomanip> #include <string> using namespace std; // ─── CONSTANTES ────────────────────────────────────────────────── const int MAX_EST = 30; const int NUM_NOTAS = 4; // ─── ARREGLOS GLOBALES (arreglos paralelos) ─────────────────────── string cedulas [MAX_EST]; string nombres [MAX_EST]; double notas [MAX_EST][NUM_NOTAS]; double promedios[MAX_EST]; bool activo [MAX_EST]; int totalEst = 0; // ─── PROTOTIPOS ────────────────────────────────────────────────── // [aquí van todos los prototipos según tu tabla de diseño]
A
Implementar utilitarios primero
Codifica en orden: cedulaValida → leerCedula → leerNota → separador. Cada función debe estar funcionando y probada antes de escribir la siguiente. Prueba cedulaValida con: "0123456789" (válida), "012345678" (9 dígitos → inválida), "01234a6789" (letra → inválida).
B
Módulo Estudiantes completo
Implementa en orden: buscarPorCedula → registrarEstudiante → listarEstudiantes → eliminarEstudiante. Después de registrar tus 5 estudiantes de prueba del F1, verifica que listarEstudiantes los muestra correctamente. Prueba el caso de cédula duplicada y el sistema lleno.
📌 Estrategia: un main() temporal de prueba

Durante el desarrollo, usa un main() temporal que llame directamente las funciones que estás probando, sin el menú completo. Así puedes verificar cada módulo aisladamente. Una vez que un módulo funciona, intégralo al menú real. Esta técnica —probar en aislamiento antes de integrar— es fundamental en el desarrollo profesional.

F3
Sesión 2 — Módulo de Notas y Ordenamiento
Sesión 2 · 0 – 100 min
A
Módulo de Notas
Implementa: leerNota → ingresarNotas → calcularPromedio → clasificarRendimiento → recalcularTodosPromedios. Verifica con los 5 estudiantes del F1: ingresa sus notas y comprueba que el promedio calculado coincide con tu cálculo manual. La diferencia debe ser cero o menor a 0.01 (error de punto flotante).
B
Módulo de Ordenamiento
Implementa: intercambiarEstudiantes → ordenarPorPromedio → ordenarPorNombre. CRÍTICO: después de intercambiarEstudiantes, verifica que el listado muestra todos los datos correctos para cada estudiante. Un error en esta función puede corromper todos los datos del sistema sin que el programa falle visiblemente.
C
fichaEstudiante con barra de progreso
Implementa la ficha completa con las 4 notas, promedio, clasificación y barra ASCII visual. Para la barra: int barras = (int)promedio; imprime barras bloques █ y 10-barras bloques ░. Prueba con promedio 0.0, 5.0, 7.5, 9.8 y 10.0.
C++ · intercambiarEstudiantes — swap de TODOS los arreglos paralelos
// ¡Crítico! Si olvidas uno de los arreglos, los datos quedan inconsistentes void intercambiarEstudiantes(int i, int j) { swap(cedulas[i], cedulas[j]); // ① cédula swap(nombres[i], nombres[j]); // ② nombre swap(promedios[i], promedios[j]); // ③ promedio swap(activo[i], activo[j]); // ④ estado for (int k = 0; k < NUM_NOTAS; k++) swap(notas[i][k], notas[j][k]); // ⑤ todas las notas }
🔴 Caso de prueba obligatorio para ordenamiento

Registra 5 estudiantes con notas variadas. Ordena por promedio. Verifica que el primero de la lista tiene el promedio más alto. Luego consulta la ficha del que era el segundo en orden original y confirma que sus notas siguen siendo las suyas (no las del estudiante que tomó su posición). Este es el error más común con arreglos paralelos.

F4
Sesión 2 — Módulo de Reportes (100–180 min)
Sesión 2 · 100 – 180 min
A
reporteEstadistico
Implementa el reporte con: total activos, promedio general, nota mínima, nota máxima, cantidad y porcentaje de aprobados (promedio ≥ 7.0). Verifica manualmente con tus 5 estudiantes: calcula a mano el promedio general y compara con el del programa. El error máximo aceptable es 0.01.
B
reporteTop3 e histogramaRendimiento
El Top 3 debe trabajar sobre una copia temporal de índices ordenada, sin modificar los arreglos globales. Verifica que después de llamar reporteTop3 el listado sigue en el mismo orden que antes. El histograma debe mostrar barras proporcionales al número de estudiantes en cada categoría A/B/C/D/F.
C
Menú principal con sub-menú de Reportes
Integra todos los módulos en el menú principal. El sub-menú de Reportes (opción 8) debe ser un do-while anidado. Verifica el flujo completo: registrar → ingresar notas → listar → ordenar → ver reportes → eliminar → verificar listado. Este flujo completo es la prueba de integración final.
F5
Sesión 3 — Integración Final y Análisis IA
Sesión 3 · 0 – 120 min
A
Compilación limpia con -Wall -Wextra
Compila con: g++ -Wall -Wextra -std=c++17 -o sistema gestion.cpp. Elimina todos los warnings, incluyendo los de variables no usadas y comparaciones de tipos. Un warning ignorado puede ocultar un bug real. Documenta qué warnings encontraste y cómo los corregiste.
B
Prueba de integración con escenario completo
Ejecuta el flujo completo con los 5 estudiantes del F1: (1) Registrar todos → (2) Ingresar notas → (3) Listar → (4) Ordenar por promedio → (5) Ver Top 3 → (6) Ver histograma → (7) Buscar uno por cédula → (8) Eliminar uno → (9) Verificar listado sin el eliminado → (10) Ver reporte estadístico actualizado. Documenta el resultado de cada paso.
C
Auditoría IA — análisis de alternativas
Muestra tu función ordenarPorPromedio a la IA y pregunta: "¿Cuáles son las alternativas a burbuja para ordenar arreglos paralelos en C++ básico?" Analiza las alternativas propuestas: ¿cuáles requieren conocimiento más avanzado? ¿hay alguna que puedas implementar ahora? Documenta tu análisis con al menos 200 palabras.
🤖 Uso responsable de IA en el proyecto final

En la defensa oral el docente puede pedirte que expliques cualquier función del sistema. Si usaste IA para generar código, debes haberlo revisado, entendido y modificado hasta que puedas explicarlo completamente. El criterio no es "¿escribiste cada línea?", sino "¿puedes defender cada línea?". Si hay código que no puedes explicar, reemplázalo por código que sí domines, aunque sea menos eficiente.

🎓 Nota de cierre de curso

Este proyecto marca el final de tu primer curso formal de programación. El Sistema de Gestión Académica que construiste integra todos los fundamentos: algoritmos, estructuras de control, funciones, arreglos, matrices, cadenas, búsqueda, ordenamiento y recursión. Guarda este código: en el futuro, cuando aprendas orientación a objetos, lo reescribirás usando clases y verás cuánto más elegante puede ser. Ese contraste —entre el código de hoy y el del futuro— será la mejor medida de tu crecimiento como programador.
